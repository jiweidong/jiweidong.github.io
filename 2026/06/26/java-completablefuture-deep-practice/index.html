【面试必备】CompletableFuture 深度实战：从 API 应用到源码级原理

面试官：项目中怎么处理异步任务的编排？

很多 Java 开发者在实际项目中都会遇到需要编排多个异步任务的场景——比如同时查询用户信息、订单信息和商品信息，然后聚合返回。Java 8 引入的 CompletableFuture 就是为此而生。

但很多人只是会用 supplyAsync + thenApply，遇到异常处理、超时控制、多任务聚合就不知道怎么办了。今天我们从 API 实战到源码原理，彻底搞懂 CompletableFuture。

一、从 Future 到 CompletableFuture

1.1 Future 的痛点

Future.get() 是阻塞调用，无法实现回调通知，也无法将多个 Future 编排组合。这正是 CompletableFuture 要解决的问题。

1.2 CompletableFuture 核心能力

转换 对结果做转换 thenApply, thenApplyAsync

组合 两个任务组合 thenCompose, thenCombine
异常处理 处理异常 exceptionally, handle, whenComplete
多任务聚合 等待多个任务 allOf, anyOf
超时控制 超时降级 orTimeout, completeOnTimeout (Java 9+)

三、线程池策略与最佳实践

3.1 默认线程池的问题

使用 CompletableFuture.supplyAsync(() -> {...})（不传 executor），默认使用 ForkJoinPool.commonPool()。

问题： 如果任务是 IO 密集型（HTTP 调用、数据库查询），默认线程池太小会导致任务排队，且与其他使用 commonPool 的地方共享资源。

规则： 纯计算/轻量用 non-async，IO/重量操作用 async+自定义线程池。

四、源码级原理：核心设计

4.1 核心成员变量

CompletableFuture 的核心就是一个状态机：

一旦结果被设置（result 变为非 null），状态就不可变。后续所有依赖操作直接执行。

4.2 无锁化的设计：CAS + Treiber Stack

CompletableFuture 全程不使用 synchronized 或 ReentrantLock，而是依赖 CAS 操作。

依赖管理： 使用 Treiber Stack（无锁并发栈）

为什么不用锁？ 因为 join() 可能阻塞，持有锁时阻塞会导致等待线程无法释放锁，引发性能问题。CAS 无等待、无阻塞，更适合这种场景。

4.3 thenApply 的执行流程

流程：

  1. 创建一个新的 CompletableFuture<U> 作为下游
  2. 包装 UniApply Completion，入栈
  3. 调用 tryFire() — 如果上游已完成则立即执行 fn，设置下游结果
  4. 返回下游 future

五、面试常见追问

Q1: thenApply 和 thenApplyAsync 有什么区别？

答： 区别在于执行线程。

  • thenApply 由前一个任务的线程同步执行（或立即执行如果已完成）
  • thenApplyAsync 会提交到 ForkJoinPool 或自定义线程池执行

面试官追问： 如果前一个任务在自定义线程池执行，thenApply 会在哪个线程执行？

答： 还是自定义线程池的线程。CompletableFuture 不改变执行上下文，thenApply 复用了前一个任务的执行线程。

Q2: CompletableFuture 如何处理异常？

答： 异常会被包装为 AltResult(Throwable) 存在 result 字段。下游通过 exceptionally/handle/whenComplete 来处理。
未被捕获的异常会沿着依赖链传播，直到被处理或者被忽略。

Q3: get() 和 join() 的区别？

答：

  • get() 抛 InterruptedException + ExecutionException（受检异常）
  • join() 抛 CompletionException（非受检异常）
  • 在 Stream + allOf 场景推荐 join()，代码更简洁

Q4: CompletableFuture 的线程模型有什么坑？

答： 主要坑点：

  1. 默认 commonPool() 是守护线程，主线程结束后 JVM 会退出
  2. 不传 executor 时，IO 密集型任务会阻塞 commonPool 的计算线程
  3. thenApplyAsync 每次都会重新入队，有线程切换开销
  4. 父子任务在不同线程执行时，ThreadLocal 传递需要手动处理

六、总结

CompletableFuture 是 Java 异步编程的核心工具，理解它的关键：

  1. API 分层： 创建 → 转换/消费 → 组合 → 异常处理
  2. 线程策略： IO 密集型务必自定义线程池，避免阻塞 commonPool
  3. 无锁设计： CAS + Treiber Stack 实现高效的依赖管理
  4. 异常传播： 链式传播，务必兜底处理

掌握 CompletableFuture，不仅仅是会用 API，更要理解它的线程模型和无锁设计原理，这样才能在面试和实战中游刃有余。
