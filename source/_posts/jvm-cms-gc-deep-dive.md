# 【JVM 原理】CMS 垃圾收集器深度解析：从并发标记到碎片化问题

## 前言

> 面试官："CMS 垃圾收集器的工作流程是怎样的？它有哪些优缺点？为什么在 JDK 9 中被标记为 Deprecated？"

CMS（Concurrent Mark Sweep）曾是 Java 服务端应用中最主流的垃圾收集器，它首次实现了"并发收集"这一里程碑式的特性。虽然现如今已被 G1 和 ZGC 取代，但理解 CMS 的设计思想对于掌握 JVM GC 的发展脉络依然至关重要。

---

## 一、CMS 诞生的背景

在 CMS 出现之前，HotSpot 的主要 GC 方案是：

| 收集器 | 适用场景 | 特点 |
|-------|---------|------|
| **Serial** | 单线程客户端 | Stop-The-World（STW）暂停时间长 |
| **Parallel Scavenge（PS）** | 多线程吞吐优先 | 关注吞吐量，但 GC 时仍会 STW |
| **Parallel Old** | 老年代并行收集 | Parallel Scavenge 的老年代搭档 |

这些收集器在 GC 时都会完全暂停应用线程。对于响应时间敏感的应用（如 Web 服务器、交易系统），长时间的 STW 是不可接受的。

**CMS 的核心理念：** 让 GC 线程和用户线程尽可能同时工作，最大限度地减少 STW 时间。

---

## 二、CMS 的工作流程：四次阶段详解

CMS 将 GC 过程拆分为四个阶段：

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 初始标记     │ →  │ 并发标记     │ →  │ 重新标记     │ →  │ 并发清除     │
│ (STW)       │    │ (并发)      │    │ (STW)       │    │ (并发)      │
│ 时间：极短    │    │ 时间：最长    │    │ 时间：较短    │    │ 时间：较长    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
     ↓                  ↓                  ↓                  ↓
  标记 GC Roots      追踪存活对象       修正并发期间       回收死亡对象
  直接引用的对象                        产生变化的对象       的内存空间
```

### 2.1 阶段一：初始标记（Initial Mark）—— STW

**目标：** 标记所有 GC Roots 直接引用的对象。

**什么是 GC Roots？**
- 虚拟机栈（栈帧中的本地变量表）引用的对象
- 方法区中类静态属性引用的对象
- 方法区中常量引用的对象
- 本地方法栈中 JNI 引用的对象
- 活跃线程（Thread 对象）

这一步需要 STW，但因为只标记直接引用，不遍历引用链，**时间非常短**（通常几毫秒）。

```java
// 以下对象会被初始标记标记到
public class UserService {
    private static UserCache cache = new UserCache(); // 静态引用 → GC Root
    
    public void process() {
        User user = new User();       // 局部变量 → GC Root
        Thread t = new Thread();      // 活跃线程 → GC Root
        // ...
    }
}
```

### 2.2 阶段二：并发标记（Concurrent Mark）—— 无 STW

**目标：** 从初始标记的结果出发，遍历所有存活对象。

这是 CMS 中最耗时但不需要 STW 的阶段。GC 线程和用户线程并行执行：

```
用户线程 ──[ 修改引用 ]──[ 分配对象 ]──[ 修改引用 ]──[ 分配对象 ]──
GC 线程  ─────[ 标记对象A ]────[ 标记对象B ]────[ 标记对象C ]────
```

**并发标记带来的问题：** 用户线程在 GC 标记过程中不断改变对象的引用关系，导致**并发标记三色问题**。

### 2.3 三色标记算法

在并发标记中，JVM 使用三色标记（Tri-Color Marking）算法管理对象状态：

| 颜色 | 含义 | 状态 |
|-----|------|------|
| **白色** | 未被标记，可能被回收 | 尚未访问到 |
| **灰色** | 已标记，但成员变量尚未被扫描 | 待扫描 |
| **黑色** | 自身和所有成员变量均已标记 | 安全存活 |

**并发标记可能出现的致命错误：**

假设有对象 A（黑色，标记完成）、B（灰色，待扫描）、C（白色，未标记）：

```
场景：A → B → C（A 持有 B，B 持有 C）
1. 黑色 A 已经扫描完，认为自己是安全的
2. 灰色 B 被扫描 → 发现持有 C → 准备标记 C
3. 用户线程将 A.B = C 修改为 A.B = null（B 不再引用 C）
4. 用户线程又将另外一个 D 的引用指向 C：D.C = C
5. 最终 C 的引用被切断又恢复，但 A 已经是黑色，不会再扫描
6. B 也不再引用 C，B 扫描时不会标记 C
7. 结果：C 存活但被当作垃圾回收 → **程序崩溃**
```

**CMS 的解决方案——增量更新（Incremental Update）：**

当黑色对象插入了一个指向白色对象的新引用时，将这个**黑色对象重新标记为灰色**（放入待扫描集合）。

```
A(黑) → 插入 C(白) 引用 → A 被标记为灰色 → 重新扫描 A
```

> 这就是为什么 CMS 第三个阶段需要**重新标记（Remark）**。

### 2.4 阶段三：重新标记（Remark）—— STW

**目标：** 修正并发标记期间因用户线程继续运行而产生的变动。

这个阶段需要 STW，但**比并发标记快得多**（通过增量更新只需要重新扫描被标记为灰色的对象）。

**优化技巧：** 在 Remark 前触发一次 Young GC（`-XX:+CMSScavengeBeforeRemark`），减少需要扫描的对象数量，加快 Remark 速度。

### 2.5 阶段四：并发清除（Concurrent Sweep）—— 无 STW

**目标：** 回收被标记为死亡的对象所占用的内存空间。

这一阶段 GC 线程与用户线程并发执行，**不 STW**。

但注意：CMS 采用的"标记-清除"算法（Mark-Sweep），**不是"标记-整理"（Mark-Compact）**，因此会产生**内存碎片**。

---

## 三、CMS 的内存碎片问题

### 3.1 碎片产生的原因

```
初始内存布局：[对象A(16KB)][对象B(8KB)][对象C(20KB)][空闲]
                                                      
并发清除后(回收了A和C)：[空闲(16KB)][对象B(8KB)][空闲(20KB)]
                                                       
第三次分配时(需要25KB的连续空间)：❌ 无法分配（没有足够大的连续空间）
```

清除后剩余的空间是**不连续**的，就像一个千疮百孔的瑞士奶酪。

### 3.2 碎片引发的后果

当需要分配一个大对象时，虽总空闲空间足够，但没有连续空间可用。此时 JVM 会：

1. 触发 **Full GC**（Serial Old 收集老年代，STW 时间很长）
2. 进行标记-整理，压缩碎片
3. 应用暂停时间暴增

### 3.3 碎片化监控与缓解

**监控方式：** 查看 GC 日志中的碎片率

```
# JVM 参数，打印 GC 详情
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log

# 日志中关注 Promotion Failed 和 Concurrent Mode Failure
2026-07-21T08:15:30.123+0800: [Full GC (Allocation Failure) ...
```

**核心参数：**

| 参数 | 作用 |
|-----|------|
| `-XX:+UseCMSCompactAtFullCollection` | Full GC 时开启碎片整理（默认开启） |
| `-XX:CMSFullGCsBeforeCompaction=5` | 每 N 次 Full GC 做一次碎片整理 |
| `-XX:+CMSParallelRemarkEnabled` | 并行化 Remark 阶段（JDK 7+ 默认开启） |
| `-XX:+CMSScavengeBeforeRemark` | Remark 前触发 Young GC |

## 四、CMS 的经典故障场景

### 4.1 Concurrent Mode Failure

**现象：** GC 日志中出现 `Concurrent Mode Failure`，随后触发 Full GC（Serial Old 收集）。

**原因：** CMS 并发标记/清除期间，用户线程仍在分配对象。当老年代空间即将耗尽，但 CMS 还没有完成回收时，JVM 会暂停所有线程，使用 Serial Old 进行 Full GC。

**解决方案：**

```bash
# 提前触发 CMS（默认老年代使用率达到 92% 时触发 CMS GC）
-XX:CMSInitiatingOccupancyFraction=70
-XX:+UseCMSInitiatingOccupancyOnly  # 只使用设定的阈值，不开启动态调整
```

### 4.2 Promotion Failed

**现象：** Young GC 后的存活对象晋升到老年代时，因碎片导致分配失败。

**原因：** 老年代碎片化严重，虽有足够总空间，但无连续空间容纳晋升对象。

**解决方案：**
- 增大老年代空间（`-Xmx` 调大）
- 增大 Survivor 区大小，减少提前晋升
- 配合 `-XX:+UseCMSCompactAtFullCollection` 做碎片整理

### 4.3 CMS GC 耗时过长

**现象：** CMS 各阶段总耗时数秒，影响应用响应时间。

**排查思路：**
```
CMS GC 耗时 = 初始标记时间 + 并发标记时间 + 重新标记时间 + 并发清除时间

初始标记耗时短（几ms）→ 看 GC Roots 数量
重新标记耗时长 → 检查是否有大量对象引用变动
并发标记耗时长 → 老年代对象数量过多
并发清除耗时长 → 老年代碎片严重
```

## 五、CMS 与其他收集器的对比

| 维度 | CMS | G1 | Parallel GC | ZGC |
|-----|-----|----|------------|-----|
| **目标** | 低延迟 | 可预测暂停 | 高吞吐 | 超低延迟 |
| **算法** | 标记-清除 | 标记-整理+Region | 标记-整理 | 染色指针+读屏障 |
| **STW 时间** | 毫秒~百毫秒 | 可配置（默认 200ms） | 秒级 | <10ms |
| **并发阶段** | 并发标记+清除 | 并发标记 | 无 | 几乎全部并发 |
| **内存碎片** | ❌ 严重 | ✅ Region 级管理 | ✅ 整理 | ✅ 无 |
| **JDK 版本** | JDK 6-8(主流) | JDK 9+(默认) | 全版本 | JDK 15+(生产) |
| **堆大小** | 中小堆(4-8GB) | 中大型堆(4-64GB) | 大堆 | 超大堆(64GB+) |

## 六、CMS 被废弃的真正原因

JDK 9 将 CMS 标记为 `@Deprecated`，JDK 14 正式移除，根本原因包括：

1. **碎片化问题**：标记-清除算法天生碎片化，长运行后会频繁触发 Full GC
2. **CPU 敏感**：并发标记阶段与用户线程争抢 CPU，在高负载下性能下降明显
3. **停顿不可控**：Concurrent Mode Failure 导致无法预期的长暂停
4. **G1 成熟**：G1 提供了更好的停顿预测能力和 Region 级的内存管理

**迁移建议：**

```
JDK 8 + CMS (-XX:+UseConcMarkSweepGC)
    ↓
JDK 11 + G1 (默认GC，设置 MaxGCPauseMillis=200)
    ↓
JDK 17 + ZGC (-XX:+UseZGC) 或 Shenandoah
```

## 面试常问追问

**Q：CMS 的增量更新（Incremental Update）和 G1 的 SATB（Snapshot At The Beginning）有什么区别？**
A：CMS 在并发标记时发现黑色对象插入新引用会标记该对象为灰色，称为增量更新。G1 采用 SATB，在并发标记开始时拍快照，只记录并发期间被**删除**的引用。SATB 更高效（不需要重新扫描所有对象），但会产生浮动垃圾。

**Q：CMS 日志中看到 "promotion failed" 怎么办？**
A：通常是碎片导致。① 调大 `-XX:CMSInitiatingOccupancyFraction` 降低触发阈值，提前 GC；② 增大老年代空间；③ 考虑升级到 G1。

**Q：为什么说 CMS 对 CPU 资源敏感？**
A：并发阶段 GC 线程和用户线程共享 CPU，如果应用本身已接近 CPU 瓶颈，CMS 会进一步拖慢业务处理，导致生产消费失衡。

**Q：生产环境 JDK 8 还在用 CMS，要不要升级？**
A：如果 GC 停顿可以接受（几十毫秒），且堆不大，可以继续用。但如果堆 > 6GB 或 GC 停顿超过 200ms，建议升级到 JDK 17+ 使用 G1 或 ZGC。
