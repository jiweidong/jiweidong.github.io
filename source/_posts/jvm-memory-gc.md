## 前言

JVM 是 Java 跨平台的基石，也是面试中的"必考题"。理解 JVM 内存模型和 GC 机制，不仅能帮你通过面试，更是解决线上 OOM、CPU 飙升、GC 频繁等问题的必备技能。

---

## 一、JVM 内存区域划分

### 1.1 整体架构

```
┌─────────────────────────────────────────────────┐
│               JVM 内存结构                        │
│  ┌───────────────────┐  ┌─────────────────────┐  │
│  │    堆 (Heap)      │  │    元空间 (MetaSpace) │  │
│  │  - 新生代          │  │  - 类元信息          │  │
│  │    · Eden          │  │  - 运行时常量池      │  │
│  │    · S0 / S1       │  │                     │  │
│  │  - 老年代          │  │                     │  │
│  └───────────────────┘  └─────────────────────┘  │
│  ┌──────────────────────────────────────────────┐ │
│  │     栈 (Stack) / 程序计数器 / 本地方法栈      │ │
│  └──────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────┘
```

### 1.2 堆（Heap）—— 最大的一块

堆是 JVM 管理的最大内存区域，**所有对象实例和数组**都在堆上分配。

```java
public class HeapDemo {
    public static void main(String[] args) {
        // 这个对象就是在堆上分配的
        byte[] data = new byte[1024 * 1024 * 100]; // 100MB
        System.out.println("分配了 100MB 堆内存");
    }
}
```

**堆的细分（分代模型）：**

```
┌────────────────────────────────────────────┐
│                  堆                         │
│  ┌──────┬────┬────┐  ┌──────────────────┐  │
│  │ Eden │ S0│ S1 │  │    老年代 (Old)    │  │
│  ├──────┴────┴────┤  │                   │  │
│  │  新生代 (Young) │  │                   │  │
│  │  Young Gen     │  │                   │  │
│  └────────────────┘  └──────────────────┘  │
└────────────────────────────────────────────┘
```

- **新生代（Young Gen）：** 存放新创建的对象，约占堆的 **1/3**
  - **Eden 区：** 大多数对象首次分配的地方
  - **Survivor 区（S0 / S1）：** 每次 Minor GC 后存活的对象在这两个区之间复制
- **老年代（Old Gen）：** 存放长期存活的对象，约占堆的 **2/3**

**默认比例（可调）：** `Eden : S0 : S1 = 8 : 1 : 1`

### 1.3 栈（Stack）—— 线程私有

每个线程都有自己独立的栈，栈中存储的是**栈帧（Stack Frame）**，每个方法调用对应一个栈帧。

```java
public class StackDemo {
    public static void main(String[] args) {
        a();
    }
    
    static void a() {
        b();  // a() 的栈帧入栈
    }
    
    static void b() {
        c();  // b() 的栈帧入栈
    }
    
    static void c() {
        // c() 的栈帧入栈 → 执行完毕出栈
    }
}
```

**栈帧包含：**
- **局部变量表：** 方法参数 + 局部变量
- **操作数栈：** 执行字节码指令的临时数据
- **动态链接：** 指向运行时常量池的引用
- **方法出口：** 方法正常/异常返回时的恢复位置

> **栈溢出：** 递归调用过深会导致 `StackOverflowError`。

### 1.4 元空间（MetaSpace）—— 替代永久代

JDK 8 开始，用**元空间**替代永久代（PermGen）：

| 对比项 | 永久代 (JDK 7) | 元空间 (JDK 8+) |
|-------|--------------|----------------|
| 位置 | JVM 堆内 | 本地内存（Native Memory） |
| 默认大小 | 固定 64MB | 无上限（取决于系统内存） |
| 溢出风险 | 容易 OOM（PermGen） | 服务器内存够大则基本不会 |
| 字符串常量池 | 永久代中 | 移到堆中 |

### 1.5 程序计数器（PC Register）

- 记录当前线程执行的字节码指令地址
- **线程私有**，每个线程都有自己的 PC
- 唯一不会出现 OOM 的区域

## 二、对象在 JVM 中的完整生命周期

```
1. new → 2. 分配内存 → 3. 初始化零值 → 4. 设置对象头 → 5. 执行构造方法
                                                            ↓
                                                 ┌───────────┴──────────┐
                                                 │  可达性分析 → 存活   │
                                                 │        ↓            │
                                                 │  不可达 → GC 回收    │
                                                 └────────────────────┘
```

### 2.1 对象的创建过程

```java
User user = new User("张三", 25);
```

1. **类加载检查：** JVM 检查 `User` 类是否已加载
2. **分配内存：** 从堆中划分一块内存
   - 指针碰撞（内存规整时）
   - 空闲列表（内存碎片化时）
3. **初始化零值：** 所有字段设为默认值（0 / null / false）
4. **设置对象头：** 存储对象元数据（GC 分代年龄、锁状态等）
5. **执行构造方法：** 调用 `<init>` 方法

### 2.2 对象的访问定位

```
通过栈上的引用访问对象：
┌──────────┐      ┌──────────────┐      ┌──────────┐
│ 栈帧     │ ──→  │ 堆中的对象   │ ──→  │ 方法区   │
│ user obj │      │ 实例数据     │      │ 类型信息 │
└──────────┘      │ 对象头       │      │ 类信息   │
                  └──────────────┘      └──────────┘
```

**两种实现方式：**
- **句柄池：** 引用指向句柄池，句柄池再指向对象实例和类型信息（稳定，但多一次寻址）
- **直接指针（HotSpot 默认）：** 引用直接指向对象，对象头中存类型指针（速度更快）

## 三、垃圾回收（GC）

### 3.1 判断对象是否可回收

#### 方法一：引用计数法（主流 JVM 已弃用）

```
objA.instance = objB
objB.instance = objA
objA = null
objB = null   → 互相引用，计数不为 0，永远不会被回收 → 内存泄漏
```

**缺点：** 无法解决循环引用问题。

#### 方法二：可达性分析（HotSpot 使用）

以 **GC Roots** 为起点，向下搜索引用链，不可达的对象即为可回收对象。

```
GC Roots 包括：
┌──────────────────────────────────────┐
│ ✓ 栈帧中的局部变量引用的对象           │
│ ✓ 静态变量引用的对象                  │
│ ✓ 本地方法栈中 JNI 引用的对象          │
│ ✓ 活跃线程（Thread）                  │
│ ✓ 被同步锁（synchronized）持有的对象    │
└──────────────────────────────────────┘

           GC Roots
          /   |    \
        A     B     C
        |     |     
        D     E     F (不可达 → 回收)
              |
              G (不可达 → 回收)
```

### 3.2 引用类型

```java
// 强引用（Strong Reference）—— 不回收
User user = new User();

// 软引用（Soft Reference）—— 内存不足时回收
SoftReference<User> softRef = new SoftReference<>(new User());

// 弱引用（Weak Reference）—— 下次 GC 就回收
WeakReference<User> weakRef = new WeakReference<>(new User());

// 虚引用（Phantom Reference）—— 任何时候都可能回收，用于对象回收跟踪
PhantomReference<User> phantomRef = new PhantomReference<>(new User(), refQueue);
```

| 引用类型 | 回收时机 | 典型用途 |
|---------|---------|---------|
| 强引用 | 永不回收（除非不可达） | 普通对象 |
| 软引用 | OOM 之前 | 缓存（如 MyBatis 缓存） |
| 弱引用 | 下次 GC | ThreadLocal、WeakHashMap |
| 虚引用 | 任何时候 | 对象回收状态的监控 |

## 四、GC 算法

### 4.1 标记-清除（Mark-Sweep）

```
初始状态：[A][B][C][D][E][F][G][H]
标记存活：[A]   [C]   [E]   [G][H]
清除后：  [A]   [C] [ ] [E] [ ] [G][H]
                    ↑ 碎片
```

**优点：** 速度快
**缺点：** 产生内存碎片，大对象分配可能触发 Full GC

### 4.2 复制（Copying）

```
Eden:[A][B][C][D][E][F]  → Minor GC
存活: [A][C][E] 复制到 S0
结果: Eden:[空] S0:[A][C][E] S1:[空]
```

**优点：** 无碎片，分配效率高
**缺点：** 内存利用率低（需要额外的 Survivor 区）

> 新生代默认用此算法，Eden:S0:S1=8:1:1，实际浪费仅 10%。

### 4.3 标记-整理（Mark-Compact）

```
标记存活：[A]   [C]   [E] [ ] [G][H]
整理后：  [A][C][E][G][H][ 连续空闲  ]
```

**优点：** 无碎片，内存利用率高
**缺点：** 需要移动对象，STW（Stop The World）时间长

> 老年代使用此算法（CMS 除外，CMS 用标记-清除）。

### 4.4 分代收集

现代 JVM 将堆分代，不同代使用不同算法：

```
对象分配 → 新生代（复制算法，Minor GC 频繁但快）
                ↓ 经过多次 GC 存活（达到年龄阈值 15）
           老年代（标记-整理算法，Major/Full GC 少但慢）
```

## 五、常见的垃圾收集器

### 5.1 新生代收集器

| 收集器 | 算法 | 特点 | 适用场景 |
|-------|------|------|---------|
| **Serial** | 复制 | 单线程，STW 暂停 | 客户端、小内存 |
| **ParNew** | 复制 | 多线程并行 | CMS 的搭档 |
| **Parallel Scavenge** | 复制 | 关注吞吐量 | 批处理、后台任务 |

### 5.2 老年代收集器

| 收集器 | 算法 | 特点 | 适用场景 |
|-------|------|------|---------|
| **Serial Old** | 标记-整理 | 单线程 | CMS 后备 |
| **Parallel Old** | 标记-整理 | 高吞吐量 | 吞吐优先场景 |
| **CMS** | 标记-清除 | 低延迟，并发 | Web 服务、响应敏感 |
| **G1（JDK 9+ 默认）** | 分区标记-整理 | 可预测停顿 | 大堆内存（>4GB） |

### 5.3 CMS 收集器工作原理

```
初始标记（STW很短） → 并发标记（耗时最长） → 重新标记（STW） → 并发清除
     ↓                   ↓                     ↓               ↓
 标记 GC Roots     遍历所有存活对象      修正并发期间      清除不可达对象
                                                    产生浮动垃圾
```

**CMS 的缺点：**
- **浮动垃圾：** 并发阶段产生的垃圾，下次才能清理
- **内存碎片：** 标记-清除算法导致，达到阈值时退化为 Serial Old
- **CPU 敏感：** 并发阶段占用 CPU 资源

### 5.4 G1 收集器（Garbage First）

G1 将堆划分为 2048 个 Region，每个 Region 可以是 Eden、Survivor、Old 或 Humongous（大对象区）。

```
┌────┬────┬────┬────┬────┬────┬────┬────┐
│ E  │ E  │ S  │ O  │ O  │ H  │ O  │ O  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ O  │ O  │ S  │ E  │ O  │ O  │
├────┼────┼────┼────┼────┼────┼────┼────┤
│ O  │ E  │ E  │ O  │ O  │ O  │ O  │ S  │
└────┴────┴────┴────┴────┴────┴────┴────┘
```

**关键参数：**
```bash
-XX:+UseG1GC                    # 使用 G1
-XX:G1HeapRegionSize=4M         # Region 大小（1MB~32MB）
-XX:MaxGCPauseMillis=200        # 目标最大停顿时间（默认 200ms）
-XX:InitiatingHeapOccupancyPercent=45  # 触发并发标记的堆占用比例
```

**G1 的优势：**
- 可预测停顿时间
- 自动进行堆的压缩，无碎片问题
- 大堆场景性能优于 CMS

## 六、GC 调优实战

### 6.1 查看 JVM 参数

```bash
# 查看所有 JVM 参数
java -XX:+PrintFlagsFinal -version

# 查看当前堆内存设置
jhsdb jinfo --pid <pid>
```

### 6.2 查看 GC 日志

```bash
# JVM 启动参数
-XX:+PrintGCDetails
-XX:+PrintGCDateStamps
-Xloggc:/path/to/gc.log
-XX:+UseGCLogFileRotation
-XX:NumberOfGCLogFiles=10
-XX:GCLogFileSize=100M
```

### 6.3 常用分析工具

| 工具 | 用途 | 命令 |
|------|------|------|
| **jps** | 查看 Java 进程 | `jps -l` |
| **jstat** | 查看 GC 统计 | `jstat -gc <pid> 1000` |
| **jmap** | 堆转储 | `jmap -dump:live,format=b,file=heap.hprof <pid>` |
| **jstack** | 线程栈 | `jstack <pid>` |
| **MAT** | 堆分析 | 解析 `heap.hprof` 文件 |
| **Arthas** | 在线诊断 | `curl -O arthas.aliyun.com/arthas-boot.jar` |

### 6.4 典型调优场景

**场景一：频繁 Full GC**

```bash
# 排查命令
jstat -gcutil <pid> 1000
# 重点关注 Full GC 次数（FGC）和耗时（FGCT）

# 常见原因及对策
# 1. 老年代空间不足 → 增大 -Xmx
# 2. 大对象直接进入老年代 → 调整 -XX:PretenureSizeThreshold
# 3. System.gc() 被代码调用 → 排查代码
# 4. 元空间不足 → 增大 -XX:MaxMetaspaceSize
```

**场景二：频繁 Minor GC**

```bash
# 通常原因是 Eden 区太小
# 调整 -Xmn 增大新生代大小
# 或者增大 -XX:SurvivorRatio
```

**场景三：GC 停顿时间过长**

```bash
# 切换到 G1 收集器
-XX:+UseG1GC -XX:MaxGCPauseMillis=200
```

### 6.5 OOM 排查步骤

```bash
# 1. 启动时保留堆转储
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump.hprof

# 2. 启动后实时监控
jstat -gcutil <pid> 3s

# 3. 分析堆转储
# 用 MAT 或 VisualVM 打开 dump 文件
# 查看 Largest Objects 和 GC Roots
```

## 七、面试高频问题

**Q1：发生过 OOM 吗？怎么排查的？**

> 有两次。第一次是接口未做分页，查全量数据 List 太大导致堆 OOM —— 加的分页 + 流式查询解决。第二次是本地缓存没设容量上限，积压大量数据 —— 改用 Guava Cache 限制最大容量 + 软引用。

**Q2：强引用、软引用、弱引用、虚引用的区别？**

> 强引用永不回收；软引用在 OOM 前回收，适合做缓存；弱引用下次 GC 就回收，ThreadLocal 的 key 就是弱引用；虚引用主要用来跟踪对象被回收的时间。

**Q3：CMS 和 G1 有什么区别？**

> CMS 基于标记-清除，会产生碎片，且对 CPU 敏感；G1 把堆分区，可预测停顿时间，自动整理内存消除碎片。JDK 9 起 G1 是默认收集器，推荐大内存场景使用。

**Q4：什么时候触发 Full GC？**

> 1. Minor GC 晋升对象大小 > 老年代剩余空间 ⇒ `Promotion Failed`
> 2. System.gc() 显式调用（建议加 `-XX:+DisableExplicitGC` 禁用）
> 3. 大对象直接在老年代分配失败
> 4. 元空间（MetaSpace）不足
> 5. CMS 并发模式失败（`Concurrent Mode Failure`）

## 八、总结

1. **堆**分新生代和老年代，对象先分配在 Eden，经历多次 Minor GC 存活后进入老年代
2. **GC 判断**使用可达性分析，GC Roots 是判定起点
3. **分代收集**：新生代复制算法（快），老年代标记-整理（少碎片）
4. **G1 是 JDK 9+ 默认收集器**，大堆场景推荐使用
5. **调优的核心**是看懂 GC 日志 + 用对工具（jstat、jmap、Arthas）

遇到 JVM 问题时，第一步永远是**保留现场**（Heap Dump、GC 日志），然后对症下药，不要经验主义。
